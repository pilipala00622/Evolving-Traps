# 归因驱动幻觉评测框架推进思路整理

基于 `reference/相关工作.md` 与 `reference/从自演化合成数据到可验证奖励强化学习_执行版.md`，这里不再复述论文，而是围绕当前框架最关键的 4 个卡点，整理一套可直接指导设计和迭代的思路。

---

## 1. 先给出一个总判断

这个框架如果要真正做出价值，核心不在于“再多加几个幻觉标签”，而在于把下面三件事连成闭环：

1. `归因体系` 要能持续吸收新失败模式，而不是一次性定义完
2. `评测流程` 要把自动化筛选、人审校准、GA 优化放在不同层级
3. `最终目标` 不是演化出复杂流程，而是让评测集本身和幻觉检测结果都越来越稳定

所以建议把整个系统定义成：

`归因体系演化 + 任务/轨迹双层验证 + 人审校准 + GA 稳定性优化`

---

## 2. 卡点一：如何确保归因体系的完善和不断更新迭代

### 2.1 不要把归因体系设计成“固定标签表”

从 4D-ARE、ReDeEP、RAGLens、Self-RAG、CRAG 这些工作里可以抽出一个共同点：

- 幻觉不是单一错误
- 幻觉既可能来自 `检索阶段`
- 也可能来自 `推理阶段`
- 还可能来自 `证据使用阶段`
- 更可能来自 `最终结论越界`

因此归因体系建议分成 3 层，而不是一层标签。

### 2.2 建议的三层归因结构

#### 第一层：错误位置

- `retrieval_error`
- `evidence_alignment_error`
- `reasoning_error`
- `final_claim_error`
- `tool_execution_error`
- `memory_error`

这一层回答的是：错误主要发生在哪个环节。

#### 第二层：错误机制

- `missing_evidence`
- `conflicting_evidence_ignored`
- `unsupported_inference`
- `temporal_staleness`
- `entity_mismatch`
- `overgeneralization`
- `fabricated_detail`
- `instruction_misfollow`

这一层回答的是：它为什么错。

#### 第三层：错误后果

- `minor_noise`
- `partial_misleading`
- `decision_blocking`
- `unsafe_action_risk`
- `attribution_break`

这一层回答的是：这个错误会造成多大的实际影响。

### 2.3 用“开放归因池”而不是封闭标签集

归因体系的更新不能靠拍脑袋扩标签，建议每轮迭代都保留一类：

- `unknown_or_new_pattern`

任何自动判不稳、多人 review 分歧大、verifier 无法覆盖的失败样本，都先进入这个池子。然后按周或按版本做一次 failure clustering：

1. 聚合高频失败样本
2. 找到是否形成稳定新模式
3. 如果稳定，再升级为正式归因标签
4. 同步补充定义、示例、边界条件和反例

这样归因体系才会不断长大，而不是越做越乱。

### 2.4 归因体系要带“定义文档 + 正反例 + 判定规则”

每个标签至少维护 4 个字段：

- `definition`
- `positive_examples`
- `negative_examples`
- `decision_rule`

否则后面的人审、自动分类器、GA fitness 都会出现语义漂移。

### 2.5 更新机制建议

可以把归因体系本身当成一个版本化对象：

- `taxonomy_v1`: 初始人工设计
- `taxonomy_v2`: 加入首轮聚类发现的新模式
- `taxonomy_v3`: 基于 reviewer disagreement 和 verifier blind spot 再修正

每次升级都要记录：

- 新增了哪些标签
- 合并了哪些标签
- 哪些标签定义被收紧或放宽
- 对历史样本是否需要回标

### 2.6 一个实用判断标准

归因体系是否足够完善，不看标签数量，而看下面 3 个指标：

1. 新来的失败样本，有没有高比例能被已有标签解释
2. 不同 reviewer 对同一样本的归因一致率是否提升
3. 用这些归因做分析时，是否能指导后续优化动作

如果标签不能指导动作，这个标签大概率只是描述，不是真正有用的归因。

## 3. 卡点二：人的 review 要在什么阶段作用

### 3.1 人审不是放在最后兜底，而是放在 3 个关键节点

建议把人的 review 分成三层。

#### 第一阶段：任务发布前的人审

作用：

- 检查任务是不是可解、可验、目标明确
- 检查 `ground_truth` 是否真的成立
- 检查该样本是否只打一个主要错误模式

这是最关键的一层，因为一旦任务设计本身错了，后面所有 rollout、评分、GA 都会被污染。

#### 第二阶段：归因体系校准期的人审

作用：

- 审核自动归因是否合理
- 处理高分歧样本
- 标注新模式样本进入 `unknown_or_new_pattern`

这一阶段建议集中在人机分歧最高的样本上，而不是均匀抽查。

#### 第三阶段：版本发布前的人审

作用：

- 做 benchmark release gate
- 看数据分布是否失衡
- 看某类错误是否被过拟合
- 看 verifier 是否被“投机取巧”绕过

这不是逐条全审，而是面向版本质量的抽样审查。

### 3.2 不建议让人审深度介入每一次轨迹打分

人的 review 应该更多承担：

- `定义边界`
- `校准规则`
- `审核疑难样本`

而不是替代自动打分器去逐条大量判分。

原因很简单：

- 成本太高
- 一致性不稳定
- 速度无法支撑 GA 或大规模 rollout

所以更合理的机制是：

`自动系统做大规模筛选，人审只处理高价值、高风险、高分歧部分`

### 3.3 可以这样插入人审

建议工作流如下：

1. 候选任务生成
2. 自动任务验证
3. 人审任务质量和归因边界
4. 放入正式 benchmark
5. rollout 生成轨迹
6. 自动 verifier 打分
7. 抽样人审 verifier 边界样本和分歧样本
8. 更新 taxonomy / verifier / 生成计划

### 3.4 哪些样本必须进人审队列

- `自动归因置信度低`
- `reviewer disagreement 高`
- `新模式候选`
- `高奖励但明显可疑`
- `GA 演化后突然大量涌现的新类型样本`

这几类样本最值得人看，因为它们直接决定体系是不是在“健康演化”。

## 4. 卡点三：GA 算法应体现在哪

### 4.1 不要把 GA 目标理解成“生成更难样本”

从你当前目标看，GA 更适合优化的是：

- `评测集稳定性`
- `检测器稳定性`
- `错误覆盖面`
- `区分度`

而不是单纯追求复杂度。

复杂度如果没有稳定性约束，最后很容易把 benchmark 演化成：

- 人也难以判定
- verifier 不稳定
- reward 噪声大
- 模型分数随机波动

这会直接削弱整个框架。

### 4.2 建议把 GA 放在“评测集选择与演化”层，而不是最终判分层

更具体地说，GA 可以做三类事：

#### 第一类：演化任务候选

基因可以来自：

- domain
- tool pattern
- context length
- evidence conflict level
- temporal sensitivity
- target attribution type

作用是生成多样任务候选。

#### 第二类：演化 benchmark 子集

不是所有生成出来的任务都进入最终 benchmark，而是用 GA 搜索一组更优的子集，使它满足：

- 标签覆盖均衡
- 难度分布合理
- 多次评测方差低
- 不同模型之间有区分度
- reviewer 一致率高

这一层其实最重要，因为它直接对应“最后评测集的稳定”。

#### 第三类：演化 verifier 压力测试样本

专门生成让 verifier 犯错的边界样本，用来测试：

- verifier 是否过宽松
- verifier 是否过严格
- 是否容易被格式噪声误伤

这会直接提升检测幻觉的稳定性。

### 4.3 GA 的 fitness 应该怎么定义

不要只看单次命中率，建议至少包含 5 个部分：

- `coverage_fitness`: 是否覆盖足够多的归因类别
- `stability_fitness`: 多次运行评分波动是否小
- `separability_fitness`: 是否能拉开不同模型的得分差
- `agreement_fitness`: 自动判分和人审一致度是否高
- `novelty_fitness`: 是否持续发现有价值的新失败模式

可以理解为：

`总 fitness = 覆盖性 + 稳定性 + 区分度 + 一致性 + 新颖性`

其中最重要的建议是把 `稳定性` 和 `一致性` 权重放高。

### 4.4 评测集稳定性具体看什么

建议至少追踪以下指标：

- 同一模型多次评测的总体得分方差
- 单个样本标签判定的一致率
- 同一样本不同 seed 下 verifier 输出的一致率
- 同类错误在不同批次中的检出率波动
- benchmark 子集版本之间的分布漂移程度

如果 GA 让这些指标更稳，那它才真的对框架有贡献。

### 4.5 检测幻觉稳定性具体看什么

建议从检测器角度再看 4 个指标：

- `precision stability`
- `recall stability`
- `attribution consistency`
- `borderline-sample robustness`

也就是：

- 面对轻微 prompt 扰动时结果是否稳定
- 面对不同证据排序时结果是否稳定
- 面对近似正确答案时是否不会误杀
- 面对多跳推理样本时是否仍能定位错误来源

这部分如果不纳入 fitness，GA 很容易把系统推向“短期有效、长期不稳”。

## 5. 卡点四：总结已有框架的创新点，并说明还能继续做什么

### 5.1 当前框架已经具备的创新点

结合现有文档，你的框架至少已经有 4 个值得强调的创新方向。

#### 创新点一：把“归因”放到幻觉评测核心，而不是只做对错判定

多数 benchmark 只告诉你模型有没有错，你这里要做的是：

- 错在什么阶段
- 错误如何形成
- 是否有证据越界
- 是否能追溯到具体机制

这比传统 accuracy-only benchmark 更有诊断价值。

#### 创新点二：把任务验证和轨迹验证拆开

这是很重要的结构优势。

很多框架把“任务质量问题”和“模型表现问题”混在一起，但你这里可以分开：

- 任务先过验证
- 轨迹再过 verifier

这样能显著降低评测噪声，也更适合后续 RL 或 GA。

#### 创新点三：把人审从“全量判分”改成“校准者”

这其实是一个很好的工程策略：

- 自动系统负责规模
- 人负责定义、边界和纠偏

如果表达清楚，这一点会比单纯说“我们有 human review”更有说服力。

#### 创新点四：把 GA 的目标放在 benchmark 演化和稳定性优化上

如果你最终把 GA 落在：

- 评测集筛选
- 覆盖性增强
- 稳定性增强
- 边界样本挖掘

那这个位置是比较新且合理的，不是把 GA 当成装饰性的搜索器。

### 5.2 还可以继续做的方向

下面这些方向都很适合继续扩展。

#### 方向一：从静态标签走向层级归因图谱

不只是给一个标签，而是记录：

- 错误位置
- 错误机制
- 证据状态
- 影响等级
- 修复建议

最后形成一个 attribution graph。

#### 方向二：加入时间维度和版本维度

很多幻觉是时间敏感的，尤其是：

- 过时知识
- 动态事实
- 时序冲突

后面可以把 `temporal attribution` 做成独立轴。

#### 方向三：从单答案评测走向多轨迹一致性评测

同一个 task 不只看一次输出，而看多次 rollout 的一致性与归因稳定性。

这会让你的 benchmark 更像“系统评测”而不是“单题考试”。

#### 方向四：做 verifier 的反作弊设计

如果未来模型学会迎合 verifier，而不是真的更可信，就会出现 reward hacking。

所以后面可以做：

- verifier ensemble
- hidden check fields
- adversarial stress cases
- cross-review verifier audit

#### 方向五：把 memory / reflection 纳入归因体系

从 MemSkill、MemRL、Generative Agents 这些工作看，很多错误并不只是当前轮推理错，而是：

- 记忆提取错
- 旧经验迁移错
- 反思总结错

如果你的框架后续面向 agent，这会成为很强的扩展点。

#### 方向六：从 benchmark 走向 training signal

一旦归因和 verifier 足够稳定，就不只是评测了，还能反过来做：

- SFT 数据过滤
- RL reward shaping
- failure-driven curriculum

这会让框架从“评测工具”升级为“训练基础设施”。

## 6. 推荐你现在就采用的整体方案

### 6.1 一个推荐的主流程

1. 定义分层归因 taxonomy
2. 生成候选任务
3. 自动任务验证
4. 人审任务和归因边界
5. 发布 benchmark 子集
6. 多次 rollout 收集轨迹
7. verifier 打分和归因
8. 人审高分歧/新模式样本
9. 用 GA 优化 benchmark 子集和边界样本
10. 更新 taxonomy、verifier 和生成器

### 6.2 你目前最值得优先补的 3 个模块

#### 第一优先级：归因 taxonomy 版本化机制

没有这个，后面所有“迭代”都会变成口头上的迭代。

#### 第二优先级：人审队列机制

要明确哪些样本进人工、为什么进、审完如何回流系统。

#### 第三优先级：GA fitness 重新定义

把优化目标从“难度/新颖度”转成“稳定性 + 区分度 + 覆盖性 + 一致性”。

## 7. 可以直接写进方案里的结论

可以把这套框架概括成一句话：

`这是一个以归因体系为核心、以人审做校准、以 verifier 做规模化判定、以 GA 优化评测稳定性的自演化幻觉评测框架。`

如果再压缩成几个最关键判断，就是：

1. 归因体系必须版本化、开放化，而不是一次性静态定义
2. 人审最重要的作用是校准任务、校准边界、发现新模式，不是全量代替自动打分
3. GA 的主要贡献应体现在 benchmark 稳定性和检测稳定性，而不是只生成更难样本
4. 当前框架的真正创新在于把归因、验证、人审、演化四件事接成了一个闭环
5. 后续最值得做的是层级归因、时间维度、多轨迹一致性、反作弊 verifier、以及向训练信号反哺

---

## 8. 一句最直接的落地建议

如果只做一个最小可行版本，建议先完成下面这个闭环：

`分层归因 taxonomy v1 -> 候选任务生成 -> 自动验证 -> 人审校准 -> 多次 rollout -> verifier 打分 -> GA 选稳定 benchmark 子集 -> 新失败模式回流 taxonomy v2`

这条链一旦跑通，你的框架就不再只是“有很多想法”，而是已经具备可持续迭代的骨架。
