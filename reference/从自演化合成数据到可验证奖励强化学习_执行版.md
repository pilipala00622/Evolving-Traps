# 从自演化合成数据到可验证奖励强化学习：执行版

基于论文《From Self-Evolving Synthetic Data to Verifiable-Reward RL: Post-Training Multi-turn Interactive Tool-Using Agents》抽离，结合当前仓库现状重组为可执行方案。

目标不是复述论文，而是把论文方法落成一条可以逐步实施的工程流程。

## 0. 总目标

我们要构建一条闭环：

1. 自动合成多样化、可验证的交互任务
2. 自动筛出高质量任务和轨迹
3. 人工复核高价值样本
4. 用通过验证的数据训练用户模型和助手模型
5. 用可执行 verifier 做奖励强化学习
6. 持续反思失败样本，迭代生成流程和奖励流程

一句话表达：

`自演化数据生成 -> 任务/轨迹验证 -> 人工复核 -> SFT -> 可验证奖励 RL -> 失败反哺生成器`

## 1. 核心思想拆解

论文里最值得保留的不是某个具体模型，而是 4 个方法论：

1. 先做“可验证的数据”，再做 RL
2. 先把 user simulator 训稳，再训 agent
3. RL 用 trajectory-level 的 outcome reward，而不是只看单步 token
4. 数据引擎必须带 reflection loop，持续从失败样本更新生成计划和评估计划

对当前仓库来说，对应关系如下：

- 你已有的 `GA + benchmark candidate` 适合承接“候选任务生成”
- 新加的 `BenchmarkValidator + 人工审核流程` 适合承接“任务验证与入库”
- 接下来需要补的是：
  - 多样化 plan 生成
  - 轨迹 rollout
  - per-instance verifier
  - user model SFT
  - verifier-based RL
  - reflection / self-evolution 闭环

## 1.5 数据集组织原则

结合当前仓库的总思路，任务池不应只按 `domain / error-type / complexity / scenario_type` 切，还应显式按答案结构分层。

建议先采用四层组织：

| 层级 | 类型 | 目标 | 风险 | 执行策略 |
|---|---|---|---|---|
| `L1` | `Numeric` | 建立最稳定的 verified 主池 | 语义歧义较低，但要防单位和格式误差 | 第一优先级 |
| `L2` | `Boolean` | 测边界判断、条件翻转、规则敏感性 | 容易被提示词猜中 | 第一优先级 |
| `L3` | `Entity Set` | 测集合闭包、枚举完整性、去重能力 | 标准答案边界容易漂移 | 强约束后推进 |
| `L4` | `Citation Set` | 测证据归因、引用对齐、来源忠实性 | 最容易引入跨文档噪声 | 第一阶段固定到文档级 |

第一批构造比例建议：

- `numeric_value`: `35%`
- `boolean_decision`: `35%`
- `entity_set`: `20%`
- `citation_set`: `10%`

这个配比体现的是“稳定优先，再逐步扩展复杂层”的策略，而不是平均覆盖所有题型。

它对后续 RL 链路的价值在于：

1. `L1/L2` 更适合先做 verifier、先做 reward 通路验证
2. `L3` 可以在 verifier 稳定后引入，用来暴露集合边界错误
3. `L4` 应最后推进，并先固定到文档级，避免 reward 噪声过早淹没训练信号

## 2. 目标系统结构

建议按 6 个层次组织：

### 层 1：Plan 层

作用：

- 生成一批互补的任务生成计划与评估计划
- 明确 domain、复杂度、tool pattern、error type、style constraint

产物：

- `plans/synthesis_plan_*.json`
- `plans/eval_plan_*.json`

最小字段建议：

```json
{
  "plan_id": "airline_boundary_v1",
  "domain": "airline",
  "task_family": "interactive_tool_use",
  "target_error_types": ["错误匹配", "缺证断言"],
  "complexity_tier": "medium",
  "tool_patterns": ["ask_user", "search", "policy_check", "write_action"],
  "style_constraints": ["真实用户口吻", "隐私信息需要多轮追问"],
  "success_criteria": ["最终状态正确", "工具顺序合理", "不违反政策"],
  "failure_taxonomy": ["task_bad", "trajectory_bad", "user_sim_bad"]
}
```

### 层 2：Task 层

作用：

- 根据 synthesis plan 生成候选任务
- 每个任务是结构化对象，不是纯文本 prompt

产物：

- `tasks/task_candidates.jsonl`

建议字段：

- `task_id`
- `plan_id`
- `user_instruction`
- `private_user_state`
- `tool_schema`
- `ground_truth_final_state`
- `expected_constraints`
- `target_error_type`
- `scenario_type`
- `answer_structure_tier`
- `verifier_mode`
- `evidence_scope`

建议新增三类关键字段：

| 字段 | 说明 | 示例 |
|---|---|---|
| `answer_structure_tier` | 四层中的哪一层 | `numeric_value` / `boolean_decision` / `entity_set` / `citation_set` |
| `verifier_mode` | verifier 如何判定 | `exact_match` / `set_match` / `citation_alignment` |
| `evidence_scope` | 证据边界范围 | `snippet` / `document` / `multi_document` |

当前阶段建议限制：

- `numeric_value` -> `verifier_mode=exact_match` 或带容差的 `numeric_match`
- `boolean_decision` -> `verifier_mode=exact_match`
- `entity_set` -> `verifier_mode=set_match`
- `citation_set` -> `verifier_mode=citation_alignment`，且 `evidence_scope=document`

### 层 3：Task Verification 层

作用：

- 验证任务本身是否合格
- 不是验证 agent 表现，而是验证“任务设计是否合理、可解、可验”

判定问题：

1. 任务是否自洽
2. `ground_truth_final_state` 是否可验证
3. 任务是否真能打到目标错误类型
4. 任务是否存在过强的非目标噪声

分层后，这一层要额外回答：

5. 该任务是否被正确归入对应答案结构层
6. 该层的标准答案边界是否足够稳定
7. 该层所要求的 verifier 是否真的可实现

输出：

- `verified_tasks.jsonl`
- `rejected_tasks.jsonl`
- `task_failure_reports.jsonl`

### 层 4：Trajectory Rollout 层

作用：

- 用 assistant policy + user simulator 跑多轮交互
- 产出完整轨迹

轨迹最小结构：

```json
{
  "trajectory_id": "traj_xxx",
  "task_id": "task_xxx",
  "assistant_model": "model_a",
  "user_model": "model_u",
  "messages": [],
  "tool_calls": [],
  "final_state": {},
  "reward": 0,
  "verifier_result": {},
  "failure_attribution": ""
}
```

### 层 5：Trajectory Verification / Reward 层

作用：

- 用 per-instance verifier 比较 `final_state` 与 `ground_truth_final_state`
- 产出 trajectory-level binary reward 或 scalar reward

论文强调的是：

- 用 outcome reward
- reward 基于执行后状态是否正确
- verifier 是任务实例级的，不是一个抽象偏好模型

建议最小版 reward：

- `1.0`: 最终状态完全匹配
- `0.0`: 不匹配

后续增强版可以拆成：

- `state_match_reward`
- `policy_compliance_reward`
- `tool_efficiency_penalty`
- `target_error_penalty`

### 层 6：Reflection / Self-Evolution 层

作用：

- 聚合 task verification 和 trajectory verification 的失败原因
- 更新 synthesis plan / eval plan

反思输出建议：

- `reflection_reports/plan_xxx_iter_k.md`
- `updated_plans/*.json`

更新方向：

- 某类任务过于简单 -> 提高复杂度 / 提高 tool dependency
- 某类 verifier 太宽松 -> 收紧检查规则
- 某类 user simulator 经常出错 -> 把这类样本喂回 user SFT
- 某类 error-type 泄漏严重 -> 调整 task schema / context 结构

## 3. 执行流程

下面这条流程是建议的实际推进顺序。

### Phase A：先把“任务可验证”做起来

#### A1. 统一任务 schema

执行目标：

- 把当前 `BenchmarkItem` 再往前扩成 `TaskSpec`
- 把任务、验证结果、人工审核结果、轨迹分开建模

完成标准：

- 每个任务都带 `ground_truth_final_state`
- 每个任务都能明确对应一个 `plan_id`
- 每个任务都带 `answer_structure_tier`
- 每个任务都能映射到明确的 `verifier_mode`

#### A1.5 先建立分层主池

在真正扩大任务规模之前，先建立四层中的主池和次级池：

- 主池：`Numeric + Boolean`
- 受控扩展池：`Entity Set`
- 慢速推进池：`Citation Set(document-level only)`

阶段目标不是把四层都铺满，而是先验证：

1. `L1/L2` 是否确实最稳定
2. `L3` 的边界噪声是否可控
3. `L4` 在文档级是否足够可验

第一批样本比例沿用：

- `35% numeric_value`
- `35% boolean_decision`
- `20% entity_set`
- `10% citation_set`

#### A2. 生成候选任务

当前仓库可以先沿用：

- `main.py`
- `core/evolution.py`
- `core/benchmark_validation.py`

执行命令：

```bash
python3 main.py --mock --build-benchmark --validation-repeats 2
```

当前已有产物：

- `benchmark_candidates.json`

#### A3. 人工复核任务质量

执行命令：

```bash
python3 review_benchmark_candidates.py export \
  --input benchmark_candidates.json \
  --output benchmark_review_tasks.jsonl
```

审核完成后回写：

```bash
python3 review_benchmark_candidates.py merge \
  --input benchmark_candidates.json \
  --reviews benchmark_review_tasks.reviewed.jsonl \
  --output benchmark_candidates.reviewed.json \
  --approved-output benchmark_release_candidates.json
```

完成标准：

- 至少得到一批人工 `approve` 的单目标 error-type 任务

### Phase B：把 verifier 做成“可执行奖励”

#### B1. 为每个任务生成 verifier

执行目标：

- 每个任务都生成一个程序化检查器
- 输入：trajectory / final_state
- 输出：reward + structured failure reason

建议目录：

- `verifiers/{task_id}.py`
- `verifiers/{task_id}.json`

最小接口建议：

```python
def verify(final_state: dict, ground_truth_final_state: dict) -> dict:
    return {
        "success": True,
        "reward": 1.0,
        "matched_fields": [],
        "mismatched_fields": [],
        "failure_reason": ""
    }
```

分层视角下，第一版 verifier 优先级应为：

1. `numeric_value`
2. `boolean_decision`
3. `entity_set`
4. `citation_set(document-level)`

不要一开始就追求统一 verifier。更稳的做法是每层先有一套简单但可信的规则：

- `Numeric`: 数值完全匹配或带容差匹配
- `Boolean`: 二元真值匹配
- `Entity Set`: 集合相等、允许顺序无关
- `Citation Set`: 引文是否都能在指定文档中定位并支撑对应陈述

#### B2. 验证 verifier 本身

必须做：

- 正样本应通过
- 负样本应失败
- verifier 不应因格式噪声过度误杀

完成标准：

- 每个 verifier 至少有一组 `gold pass / gold fail` 测试样例

### Phase C：先训 user simulator，再训 agent

这是论文里非常关键的一点。

#### C1. 先准备用户轨迹 SFT 数据

数据来源：

- 通过验证的 synthetic dialogues
- 人工复核通过的高质量多轮样本

训练目标：

- 用户模型能稳定执行自己的角色
- 不乱用工具
- 能按私有状态回应，不泄露不该提前暴露的信息

完成标准：

- 在固定评测任务上，user simulator 的行为稳定
- 同任务多次 rollout 方差明显下降

#### C2. 再训练 assistant SFT

输入：

- 通过 verifier 和人工审核的数据

目标：

- 先让 agent 学会任务结构和工具使用基本套路

完成标准：

- SFT 后基础成功率明显高于 base model

### Phase D：做 verifier-based RL

#### D1. RL rollout

每个 task 采样 `G` 条 trajectory，核心是做 group-relative comparison。

建议记录：

- `task_id`
- `group_id`
- `trajectory_id`
- `reward`
- `final_state`
- `verifier_result`

#### D2. Dynamic filtering

论文强调：

- 如果一组轨迹全成或全败，优势全为 0，没有学习信号
- 这些 task 应从该批训练里过滤掉

执行规则：

- 保留 reward 有差异的任务组
- 丢弃全 1 或全 0 的组

#### D3. 大 batch 稳定训练

论文结论之一：

- prompt 数和每个 prompt 的 trajectory 数如何分配不是最关键
- 更关键的是总 batch size 足够大，保证 advantage 更稳定

执行建议：

- 从小 batch 做通路验证
- 真训练时优先拉大总 rollout 数

### Phase E：把失败喂回生成器

每一轮训练后都做失败归因聚合：

1. 是 task 本身设计不好
2. 是 trajectory 生成不好
3. 是 user simulator 错
4. 是 verifier 太松或太严

然后更新：

- synthesis plan
- eval plan
- user SFT 数据
- verifier 规则

## 4. 人工必须参与的环节

论文强调“减少人工”，不是“不要人工”。下面这些地方必须人工 check。

### 必须人工 check 的部分

1. 目标 error-type 标注
2. `ground_truth_final_state` 是否正确
3. verifier 的通过/拒绝规则是否合理
4. user simulator 是否忠实扮演用户
5. `real_time / out_of_date` 时间元数据是否真实
6. 奖励函数是否和真实目标一致

### 必须人工标注的最小字段

- `labeled_target_error_type`
- `is_single_target_error`
- `reference_answer_supported`
- `final_state_is_correct`
- `verifier_is_valid`
- `time_metadata_correct`
- `release_decision`

## 5. 论文结论转成工程决策

这篇论文最值得直接转成工程决策的点如下：

### 决策 1

不要直接上 RL，先做高质量可验证数据。

### 决策 2

不要直接拿 base model 当 user simulator。

先做 user SFT，再让它进入 RL rollout。

### 决策 3

奖励要尽量来自程序 verifier，而不是模糊偏好分。

### 决策 4

数据生成器不能是一次性脚本，必须持续吸收失败样本。

### 决策 5

最终比较的是 trajectory outcome，不是单步看起来是否像正确。

### 决策 6

不要把所有 factual task 混成一个池。先按答案结构分层，优先建立稳定层的主池，再把复杂层作为受控扩展层逐步纳入。

## 7. 推荐执行顺序

如果要真落地，不建议同时铺开所有模块。建议按下面顺序推进。

### 第一阶段

- 完成任务 schema
- 完成任务 verifier
- 完成人工审核闭环

验收标准：

- 能稳定产出人工批准的 benchmark task

### 第二阶段

- 生成轨迹
- 生成 per-instance verifier
- 做离线 reward 计算

验收标准：

- 给定一条 trajectory，系统能稳定输出 reward 和失败原因

### 第三阶段

- 训练 user model SFT
- 训练 agent SFT

验收标准：

- user rollout 方差降低
- agent SFT 过 base model

### 第四阶段

- 上 verifier-based RL
- 加 dynamic filtering
- 加 reflection loop

验收标准：

- RL 明显优于 SFT
- 新失败样本能反向改善数据生成

## 9. 成功标准

这个方案真正跑通的标志，不是“能生成很多样本”，而是同时满足：

1. 生成的数据能稳定通过任务级验证
2. 人工审核通过率逐轮提升
3. user simulator 足够稳定，不再主导 reward 噪声
4. verifier reward 能可靠区分好坏 trajectory
5. RL 相比 SFT 继续带来增益
6. 失败样本能反向推动下一轮生成质量提升

## 10. 一句话总结

论文真正可执行的主线是：

`多样化计划 -> 结构化任务 -> 任务验证 -> 轨迹生成 -> verifier 奖励 -> 先训 user 再训 agent -> 动态过滤 -> 失败反思 -> 自演化闭环`

对当前仓库来说，最现实的落地路线是：

`先把任务验证和人工审核做实 -> 再补轨迹和 verifier -> 最后上 user SFT + agent RL`
