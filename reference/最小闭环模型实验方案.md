# 最小闭环模型实验方案

这份文档对应你确认的最小可行闭环：

`分层归因 taxonomy v1 -> 候选任务生成 -> 自动验证 -> 人审校准 -> 多次 rollout -> verifier 打分 -> GA 选稳定 benchmark 子集 -> 新失败模式回流 taxonomy v2`

目标不是一次做完整训练系统，而是先把“模型实验”跑通，证明这个闭环能稳定地产生：

1. 可解释的幻觉归因结果
2. 可重复的 benchmark 稳定性结论
3. 能反馈回 taxonomy 的新失败模式

---

## 1. 实验总目标

最小闭环版本建议回答 4 个实验问题：

### Q1. 候选任务经过自动验证和人审后，是否能形成一批“可发布”的高质量样本？

也就是验证：

- 自动验证是否有用
- 人审是否能有效筛掉混合题、伪难题、不可验证题

### Q2. 同一个模型在同一批任务上多次 rollout，幻觉检测结果是否稳定？

也就是验证：

- verifier 是否稳定
- 任务设计是否稳定
- 不是只靠一次随机结果

### Q3. GA 选出的 benchmark 子集，是否比原始候选集更稳定、更有区分度？

也就是验证：

- GA 不是只在“生成新题”
- 而是真的优化 benchmark 质量

### Q4. 失败样本是否能反哺 taxonomy，形成更好的 v2？

也就是验证：

- taxonomy 不是静态标签表
- 新失败模式能被系统性吸收

---

## 2. 这轮实验不追求什么

最小版本先不要追求：

- 直接做 RL 训练
- 复杂多 agent 协作
- 大规模人审
- 一次性覆盖所有 error type
- 一开始就做特别复杂的时间型题

这一轮只要求把闭环跑通，证明框架结构成立。

---

## 3. 实验对象和范围

### 3.1 先选少量 error type

建议第一轮只选 3 类，优先选当前仓库和相关工作里最容易形成清晰边界的类型：

- `引用错误`
- `错误拼接`
- `缺证断言`

原因：

- 都和证据支撑直接相关
- verifier 相对更容易设计
- 人审边界也比“泛化推理错”更容易统一

### 3.2 先选少量 domain

建议先选 2 到 3 个领域：

- `健康医疗`
- `法律政务`
- `科技互联网`

原因：

- 都具备较强事实支撑需求
- 幻觉的危害和归因更容易讲清楚

### 3.3 先限定场景

建议第一轮先只跑：

- `static`

等最小闭环稳定后，再扩展：

- `real_time`
- `out_of_date`

---

## 4. 实验分阶段设计

整个实验建议分成 5 个阶段。

---

## 5. Phase A：taxonomy v1 建立

### 5.1 目标

先建立一个足够小、足够清晰、可用于标注和分析的归因体系 v1。

### 5.2 建议结构

先用三层结构：

- `错误位置`
- `错误机制`
- `错误后果`

示例：

```json
{
  "error_location": "final_claim_error",
  "error_mechanism": "unsupported_inference",
  "impact_level": "partial_misleading"
}
```

### 5.3 最小标签集建议

错误位置：

- `retrieval_error`
- `evidence_alignment_error`
- `reasoning_error`
- `final_claim_error`

错误机制：

- `missing_evidence`
- `conflicting_evidence_ignored`
- `unsupported_inference`
- `fabricated_detail`

错误后果：

- `minor_noise`
- `partial_misleading`
- `decision_blocking`

保留一个开放桶：

- `unknown_or_new_pattern`

### 5.4 输出物

建议新增一个手工维护文件，例如：

- `reference/taxonomy_v1.md`

里面至少包含：

- 标签定义
- 正例
- 反例
- 判定规则

### 5.5 验收标准

- reviewer 对同一标签的理解一致
- 不需要很多标签，但定义必须清楚

---

## 6. Phase B：候选任务生成 + 自动验证

### 6.1 目标

生成第一批候选任务，并用现有自动验证流程做粗筛。

### 6.2 当前仓库可直接使用的命令

```bash
python3 main.py --mock --build-benchmark --validation-repeats 3
```

如果要换真实模型：

```bash
python3 main.py \
  --build-benchmark \
  --generation-model gpt-5.1 \
  --judge-model gpt-5.1 \
  --eval-models gpt-5.2 deepseek-v3.2 ernie-5.0 \
  --validation-repeats 3
```

### 6.3 本阶段关键输出

当前仓库已经会导出：

- `benchmark_candidates.json`
- `task_specs.json`
- `verifier_specs.json`
- `trajectory_specs.json`
- `updated_plans.json`

### 6.4 本阶段要记录的指标

- 候选题总数
- 每个 error type 的候选数量
- 每个 plan 的候选数量
- target trigger rate
- non-target leakage
- answerability
- naturalness
- recommended_for_release 比例

### 6.5 本阶段的实验问题

重点看：

- 自动验证是否能把明显不可用题筛掉
- 各 error type 的自动统计是否有明显失衡
- 哪些 plan 反复生成混合题

### 6.6 验收标准

- 每个目标 error type 至少有一批候选题
- 不是所有题都落入 `needs_review`
- 自动统计能初步区分“相对可用”和“明显不可用”的候选

---

## 7. Phase C：人审校准

### 7.1 目标

把自动候选题变成一批真正可用于实验的 release 候选集。

### 7.2 当前仓库可直接使用的命令

导出待审样本：

```bash
python3 -m human_review.cli export \
  --input benchmark_candidates.json \
  --output human_review/data/tasks/benchmark_review_tasks.jsonl
```

回写审核结果：

```bash
python3 -m human_review.cli merge \
  --input benchmark_candidates.json \
  --reviews human_review/data/reviews/benchmark_review_tasks.reviewed.jsonl \
  --output human_review/data/releases/benchmark_candidates.reviewed.json \
  --approved-output human_review/data/releases/benchmark_release_candidates.json
```

### 7.3 人审时必须判断的字段

- `confirmed_target_error_type`
- `reference_answer_supported`
- `final_state_is_correctly_specified`
- `verifier_design_is_feasible`
- `reward_should_be_verifiable`
- `query_is_natural`
- `is_single_target_error`

### 7.4 本阶段要记录的指标

- approve rate
- revise rate
- reject rate
- 单目标错误样本占比
- verifier 可行样本占比
- reviewer disagreement rate

### 7.5 本阶段实验结论怎么写

重点不是“通过了多少”，而是：

- 自动验证和人审之间的一致性如何
- 自动验证错在哪些地方
- 哪些错误类型最容易混淆
- 哪些任务虽然看起来难，但其实不可验证

### 7.6 验收标准

- 得到一份 `benchmark_release_candidates.json`
- 至少形成一批 `单目标错误 + verifier 可行 + 标答可支撑` 的题

---

## 8. Phase D：多次 rollout + verifier 打分

### 8.1 目标

验证“检测结果是否稳定”。

这一阶段是整个最小闭环最关键的实验部分。

### 8.2 实验对象

从 `benchmark_release_candidates.json` 中选择：

- 每个 error type 20 到 30 题
- 每个 domain 尽量均衡

如果样本还不够，就每类先做 10 题的小实验。

### 8.3 被测模型建议

第一轮建议选 3 个能力层次不同的模型：

- 一个强模型
- 一个中等模型
- 一个较弱模型

例如当前仓库已有默认配置可参考：

- `gpt-5.2`
- `deepseek-v3.2`
- `ernie-5.0`

### 8.4 rollout 设计

对每个 `task_id`、每个 `model`，重复 rollout `K=5` 次。

建议固定：

- 相同 task
- 相同 verifier
- 相同 user prompt 结构

只让模型采样随机性产生波动。

### 8.5 当前仓库现状

当前仓库已经导出了 `trajectory_specs.json`，但还没有完整的 rollout executor。

所以最小版可以这样做：

1. 从 `trajectory_specs.json` 读取任务初始状态
2. 用一个简单脚本循环调用 assistant model
3. 将最终输出映射成 `final_state`
4. 根据 `verifier_specs.json` 执行规则匹配
5. 产出 `reward` 和 `failure_reason`

### 8.6 建议新增的最小实验脚本

建议后续补一个最小脚本：

- `run_minimal_rollout_experiment.py`

它的输入：

- `human_review/data/releases/benchmark_release_candidates.json`
- `trajectory_specs.json`
- `verifier_specs.json`

它的输出：

- `experiment_outputs/rollout_results.jsonl`

每条至少包含：

```json
{
  "task_id": "xxx",
  "model_name": "gpt-5.2",
  "run_id": 3,
  "final_state": {},
  "reward": 1.0,
  "failure_reason": "",
  "predicted_attribution": {},
  "latency_ms": 0
}
```

### 8.7 本阶段核心指标

按 task 看：

- pass rate
- reward variance
- attribution consistency

按 model 看：

- average reward
- hallucination detection rate
- per-error-type success rate
- same-task repeat consistency

按 benchmark 看：

- overall score variance
- per-type variance
- borderline sample disagreement

### 8.8 这一步最关键的判断

不是只看谁分高，而是看：

- 同一模型在同一任务上重复多次，结果是否稳定
- 不同模型之间能否被稳定区分开
- 某些题是不是对所有模型都随机

### 8.9 验收标准

- 同一模型多次跑同一批任务，总分波动可接受
- verifier 能稳定输出 reward 和 failure reason
- 至少能观察到模型间差异

---

## 9. Phase E：GA 选稳定 benchmark 子集

### 9.1 目标

用 GA 从 release candidates 中选择更稳、更有区分度的一组 benchmark 子集。

### 9.2 这一阶段为什么重要

因为最终不是所有过审题都应该进正式 benchmark。

有些题虽然“合法”，但：

- 方差太大
- 对所有模型都一样
- reviewer 也经常分歧
- verifier 容易误判

这些都应该被降权或剔除。

### 9.3 GA 的个体定义

这一步建议不要再让 GA 生成题，而是让它选择题目子集。

一个个体就是：

- 一个 benchmark 子集

基因可以是：

- 选中哪些 task
- 每类 task 的占比
- 每个 domain 的配额

### 9.4 fitness 建议

最小版先用 4 个指标：

- `coverage`
- `stability`
- `separability`
- `agreement`

可以写成：

`fitness = 0.3 * coverage + 0.35 * stability + 0.25 * separability + 0.1 * agreement`

其中：

- `coverage`: error type / domain / complexity 是否均衡
- `stability`: 多次 rollout 分数方差是否小
- `separability`: 不同模型得分差是否大
- `agreement`: 自动判分与人审抽查是否一致

### 9.5 输入数据

GA 选子集时需要读取：

- 人审结果
- rollout 结果
- verifier 打分结果

### 9.6 输出数据

建议输出：

- `experiment_outputs/benchmark_subset_v1.json`
- `experiment_outputs/ga_fitness_report_v1.md`

### 9.7 最终要比较的不是绝对分数，而是两组集合

需要比较：

- `原始 release candidates`
- `GA 筛选后的 stable subset`

比较指标：

- 总体得分方差
- per-type 方差
- 模型间区分度
- reviewer disagreement rate

### 9.8 验收标准

- GA 选出的子集稳定性优于原始集合
- 模型区分度不下降，最好提升

---

## 10. Phase F：失败模式回流 taxonomy v2

### 10.1 目标

用实验中发现的新失败模式更新 taxonomy。

### 10.2 哪些样本要重点分析

- verifier 经常不稳定的样本
- reviewer 经常分歧的样本
- 高 reward 但人工觉得可疑的样本
- 同一 task 多次 rollout 结果高度波动的样本

### 10.3 建议分析方式

先把这些样本统一放入：

- `unknown_or_new_pattern`

然后做一轮人工聚类：

1. 哪些失败模式反复出现
2. 是否能形成清晰定义
3. 是否值得升级为正式标签

### 10.4 taxonomy v2 需要记录什么

- 新增了哪些标签
- 哪些标签被合并
- 哪些标签定义被改写
- 历史样本是否回标

### 10.5 输出物

建议输出：

- `reference/taxonomy_v2.md`
- `experiment_outputs/failure_pattern_analysis_v1.md`

### 10.6 验收标准

- taxonomy v2 不是随意扩标签
- 至少有若干新模式来自真实实验失败，而不是主观想象

---

## 11. 一套建议的实验矩阵

第一轮先做一个小矩阵就够了。

### 11.1 任务规模

- 3 个 error type
- 2 到 3 个 domain
- 每类 10 到 30 题

### 11.2 模型规模

- 3 个模型
- 每模型每题重复 5 次

### 11.3 总 rollout 量估算

如果按：

- 3 个 error type
- 每类 20 题
- 3 个模型
- 每题 5 次

总 rollout 数约为：

`3 * 20 * 3 * 5 = 900`

这是一个可控的最小实验量级。

## 13. 实验报告建议怎么写

最后建议固定成 5 个部分。

### Part 1. 数据与任务来源

说明：

- taxonomy v1 范围
- domain 和 error type 选择
- 候选任务规模

### Part 2. 自动验证和人审结果

说明：

- 自动验证筛掉了什么
- 人审最终保留了什么
- 自动和人工主要分歧在哪

### Part 3. 多次 rollout 稳定性

说明：

- 各模型总体表现
- 各 error type 表现
- 稳定性和波动性

### Part 4. GA 子集优化效果

说明：

- 原始集合 vs GA 子集
- 稳定性是否提升
- 区分度是否提升

### Part 5. taxonomy v2 的更新

说明：

- 新发现了哪些失败模式
- 为什么这些模式值得纳入 v2

## 14. 当前最建议你立刻做的事

如果只追求最小闭环落地，建议按下面顺序推进。

### Step 1

先手工确定 `taxonomy v1`，不要超过 10 个核心标签。

### Step 2

用当前仓库跑出第一版：

```bash
python3 main.py --mock --build-benchmark --validation-repeats 3
```

### Step 3

做一轮小规模人审，先拿到一批 `release candidates`。

### Step 4

补一个最小 rollout + verifier 实验脚本，先让 3 个模型各跑 5 次。

### Step 5

把 rollout 结果喂给 GA，不再进化题目，而是进化 benchmark 子集。

### Step 6

分析高波动样本和分歧样本，更新成 `taxonomy v2`。

---

## 15. 一句话总结实验主线

这套最小模型实验，不是为了直接证明“哪个模型最强”，而是为了证明：

`这个框架能稳定地产生可验证任务、稳定地检测幻觉、稳定地区分模型，并把新失败模式持续吸收到归因体系中。`
